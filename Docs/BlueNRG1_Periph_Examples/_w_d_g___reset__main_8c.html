Detailed Description

This code provides an example how the watchdog timer works and how it could be used to reset the system if the watchdog interrupt flag is not cleared within the watchdog timeout period.

ATOLLIC project

To use the project with ATOLLIC TrueSTUDIO for ARM, please follow the instructions below:

  1. Open the ATOLLIC TrueSTUDIO for ARM and select File->Import... Project menu.
  2. Select Existing Projects into Workspace.
  3. Select the ATOLLIC project
  4. Select desired configuration to build from Project->Manage Configurations
  5. Select Project->Rebuild Project. This will recompile and link the entire application
  6. To download the binary image, please connect STLink to JTAG connector in your board (if available).
  7. Select Project->Download to download the related binary image.
  8. Alternatively, open the BlueNRG1 Flasher utility and download the built binary image.

Usage

The watchdog timer is a 32-bit down counter that divides the clock input (32.768 kHz) to produce an interrupt. A watchdog interrupt is generated each time the counter reaches 0. The counter is then reloaded with the content of the WDT_LR register. If the interrupt status flag is not cleared and a new interrupt is generated then the watchdog could generate a reset to reboot the system.

This example demonstrates the watchdog functionality and the possibility to use it to reboot the system when the watchdog interrupt is not serviced during the watchdog period (interrupt status flag is not cleared).

The watchdog is configured to generate the interrupt with period of 15s and then it is enabled. Then the PUSH1 button, IO13 pin, state is watched. Any change of this pin triggers the watchdog counter reload and restart of the 15s period measurement.

If the IO13 pin state does not change during this 15s period the watchdog generates an interrupt that is intentionally not cleared. Therefore, the interrupt remains pending, and for that reason the watchdog interrupt service routine is called continuously, the system is stuck in the watchdog interrupt handler. It can no more proceed with the user code execution and this reset the chip. Once the watchdog timeout occurs, the second time the system reboot is triggered as a new watchdog interrupt is generated in the time when the previous interrupt is still pending. Then the application starts again to measure the 15s period.

The three user LEDs are toggled with increased rates until the board is reset. The user must press the PUSH1 button to avoid the reset of the board. Once the PUSH1 button is pressed, the LEDs toggling speed is restored with the 15s watchdog timer.
